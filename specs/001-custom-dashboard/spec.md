# Feature Specification: 커스텀 대시보드

**Feature Branch**: `001-custom-dashboard`
**Created**: 2026-04-02
**Status**: Draft

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 위젯 추가 및 삭제 (Priority: P1)

사용자는 제공된 위젯 목록에서 원하는 위젯을 선택해 대시보드에 추가하고, 불필요한 위젯은 삭제하여 자신만의 업무 화면을 구성할 수 있다.

**Why this priority**: 위젯 추가/삭제는 모든 대시보드 기능의 전제 조건이다. 이 기능 없이는 배치, 크기 조절, 저장 기능도 의미가 없다.

**Independent Test**: 위젯 목록 패널에서 위젯을 추가하고, 대시보드에 나타나는 것을 확인한 뒤, 삭제 버튼으로 제거되는 것을 확인함으로써 독립적으로 검증 가능하다.

**Acceptance Scenarios**:

1. **Given** 대시보드가 비어있고 위젯 목록이 표시될 때, **When** 사용자가 특정 위젯을 추가하면, **Then** 해당 위젯이 대시보드의 첫 번째 사용 가능한 위치에 나타난다.
2. **Given** 동일한 종류의 위젯이 이미 대시보드에 있을 때, **When** 같은 위젯을 다시 추가하면, **Then** 중복 위젯이 별도 인스턴스로 추가된다.
3. **Given** 대시보드에 위젯이 존재할 때, **When** 사용자가 위젯의 삭제 버튼을 누르면, **Then** 해당 위젯만 제거되고 나머지 위젯은 유지된다.

---

### User Story 2 - 위젯 드래그 앤 드롭 배치 (Priority: P2)

사용자는 대시보드에 추가된 위젯을 드래그하여 원하는 위치로 자유롭게 이동시켜 화면 구성을 최적화할 수 있다.

**Why this priority**: 위젯을 추가한 후 위치를 조정하는 것이 커스텀 대시보드의 핵심 가치다. 고정 레이아웃과의 차별점이다.

**Independent Test**: 하나의 위젯을 드래그하여 다른 위치에 놓았을 때 그리드 스냅에 맞게 배치되는지 확인함으로써 독립적으로 검증 가능하다.

**Acceptance Scenarios**:

1. **Given** 위젯이 대시보드에 배치되어 있을 때, **When** 사용자가 위젯을 드래그하여 다른 영역에 놓으면, **Then** 위젯이 가장 가까운 그리드 셀에 스냅되어 배치된다.
2. **Given** 두 위젯이 같은 위치에 겹치려 할 때, **When** 사용자가 드래그로 위젯을 이동하면, **Then** 기존 위젯은 자동으로 밀려나 충돌을 피한다.
3. **Given** 드래그 중인 위젯이 있을 때, **When** 사용자가 마우스를 놓으면, **Then** 위젯이 해당 위치에 고정되고 드래그 인디케이터가 사라진다.

---

### User Story 3 - 위젯 크기 조절 (Priority: P2)

사용자는 각 위젯의 크기를 조절 핸들을 통해 중요한 정보는 크게, 덜 중요한 정보는 작게 표시되도록 조정할 수 있다.

**Why this priority**: 크기 조절은 레이아웃의 완성도를 높이는 기능으로, 위젯 추가와 함께 핵심 커스터마이징 기능이다.

**Independent Test**: 하나의 위젯의 우측 하단 핸들을 드래그하여 크기가 변경되는 것을 확인함으로써 독립적으로 검증 가능하다.

**Acceptance Scenarios**:

1. **Given** 위젯이 대시보드에 배치되어 있을 때, **When** 사용자가 우측 하단 핸들을 드래그하면, **Then** 위젯의 너비와 높이가 그리드 단위로 변경된다.
2. **Given** 위젯이 최소 크기에 도달했을 때, **When** 사용자가 계속 줄이려 하면, **Then** 더 이상 줄어들지 않고 최소 크기를 유지한다.
3. **Given** 위젯이 최대 크기에 도달했을 때, **When** 사용자가 계속 늘리려 하면, **Then** 더 이상 늘어나지 않고 최대 크기를 유지한다.

---

### User Story 4 - 레이아웃 저장 및 복원 (Priority: P1)

사용자가 커스터마이즈한 대시보드 구성(위젯 종류, 위치, 크기)이 세션 간에 유지되어, 페이지를 새로고침하거나 재방문해도 동일한 화면을 볼 수 있다.

**Why this priority**: 저장 기능이 없으면 매번 다시 설정해야 하므로 커스터마이즈 기능 자체의 가치가 사라진다.

**Independent Test**: 위젯을 배치하고 새로고침 후 동일한 구성이 유지되는지 확인함으로써 독립적으로 검증 가능하다.

**Acceptance Scenarios**:

1. **Given** 사용자가 대시보드를 커스터마이즈한 후, **When** 페이지를 새로고침하면, **Then** 이전에 설정한 위젯의 종류, 위치, 크기가 그대로 복원된다.
2. **Given** 사용자가 위젯을 삭제한 후, **When** 페이지를 새로고침하면, **Then** 삭제된 위젯은 나타나지 않는다.
3. **Given** 저장된 레이아웃이 없는 신규 사용자일 때, **When** 대시보드에 처음 접속하면, **Then** 미리 정의된 기본 레이아웃이 표시된다.

---

### User Story 5 - 반응형 레이아웃 (Priority: P3)

사용자는 데스크탑, 태블릿, 모바일 등 다양한 화면 크기에서 대시보드에 접속했을 때 각 디바이스에 최적화된 위젯 배치를 경험할 수 있다.

**Why this priority**: 반응형은 접근성 측면에서 중요하지만, 핵심 기능(추가/배치/저장) 이후에 다루는 것이 적절하다.

**Independent Test**: 동일한 대시보드를 다양한 화면 너비에서 열었을 때 위젯이 겹치지 않고 자연스럽게 재배열되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 데스크탑에서 설정한 레이아웃이 있을 때, **When** 모바일 화면 너비로 접속하면, **Then** 위젯들이 단일 열로 세로 배열되어 겹침 없이 표시된다.
2. **Given** 좁은 화면에서 대시보드를 사용 중일 때, **When** 브라우저 창을 넓히면, **Then** 위젯이 더 넓은 그리드 레이아웃으로 자연스럽게 재배열된다.

---

### Edge Cases

- 위젯 추가 시 대시보드 공간이 부족하면? → 스크롤 가능한 영역 하단에 자동 추가된다.
- 저장된 레이아웃 데이터가 손상되거나 유효하지 않은 경우? → 기본 레이아웃으로 폴백하고 사용자에게 알림을 표시한다.
- 위젯이 0개인 빈 상태에서 새로고침하면? → 빈 대시보드 상태를 저장하고 빈 화면으로 복원한다.
- 드래그 중 브라우저 창 밖으로 마우스가 나가면? → 드래그가 취소되고 위젯이 원래 위치로 돌아간다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 사용 가능한 위젯 정의 목록을 제공해야 하며, 사용자가 원하는 위젯을 선택해 대시보드에 추가할 수 있어야 한다.
- **FR-002**: 시스템은 대시보드에 배치된 위젯을 개별적으로 삭제하는 기능을 제공해야 한다.
- **FR-003**: 사용자는 대시보드에 배치된 위젯을 드래그하여 원하는 위치로 이동할 수 있어야 하며, 그리드 스냅이 적용되어야 한다.
- **FR-004**: 사용자는 각 위젯의 우측 하단 핸들을 통해 위젯의 너비와 높이를 자유롭게 변경할 수 있어야 한다.
- **FR-005**: 각 위젯은 고유한 최소 크기와 최대 크기를 가지며, 해당 범위를 벗어나는 크기 조절은 허용되지 않아야 한다.
- **FR-006**: 위젯 이동 및 크기 조절 중 다른 위젯과 충돌이 발생할 경우, 시스템은 충돌을 자동으로 해소하여 위젯 겹침을 방지해야 한다.
- **FR-007**: 사용자가 구성한 대시보드 레이아웃(위젯 종류, 위치 x/y, 크기 w/h)은 저장되어 새로고침 및 재방문 시 복원되어야 한다.
- **FR-008**: 저장된 레이아웃이 없는 신규 사용자에게는 미리 정의된 기본 레이아웃이 표시되어야 한다.
- **FR-009**: 대시보드는 화면 크기(데스크탑, 태블릿, 모바일)에 따라 위젯이 겹치지 않고 재배열되는 반응형 레이아웃을 지원해야 한다.
- **FR-010**: 저장된 레이아웃 데이터가 유효하지 않을 경우 기본 레이아웃으로 폴백하고 사용자에게 안내를 제공해야 한다.

### Key Entities

- **대시보드(Dashboard)**: 사용자 고유의 위젯 배치 공간. 사용자 1명당 1개의 대시보드를 가진다.
- **위젯 인스턴스(Widget Instance)**: 대시보드에 배치된 특정 위젯. 위젯 종류, 위치(x, y), 크기(w, h), 고유 ID를 속성으로 가진다.
- **위젯 정의(Widget Definition)**: 개발자가 사전에 등록한 위젯의 명세. 위젯 이름, 설명, 기본 크기, 최소/최대 크기를 포함한다.
- **레이아웃(Layout)**: 특정 화면 크기(breakpoint)에서의 위젯 인스턴스 배열 상태. breakpoint별로 독립적으로 저장된다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 사용자가 위젯 추가부터 원하는 위치 배치까지의 전체 작업을 60초 이내에 완료할 수 있다.
- **SC-002**: 페이지 새로고침 후 2초 이내에 이전에 저장된 레이아웃이 완전히 복원되어 표시된다.
- **SC-003**: 모바일(320px)부터 와이드스크린(1920px)까지 모든 화면 크기에서 위젯 겹침 없이 레이아웃이 표시된다.
- **SC-004**: 처음 대시보드를 사용하는 사용자의 90% 이상이 별도 안내 없이 위젯 추가 및 배치 기능을 스스로 사용할 수 있다.
- **SC-005**: 드래그 및 크기 조절 동작 중 시각적 끊김 없이 부드러운 인터랙션이 제공된다.

## Assumptions

- 이 대시보드는 로그인한 사용자를 대상으로 하며, 인증은 기존 시스템을 재사용한다.
- 레이아웃 저장은 사용자별로 서버에 영구 저장된다. (현재 PoC 단계에서는 DB 연동을 고려하지 않으며, 로컬 저장소를 임시로 사용한다.)
- 위젯의 실제 콘텐츠(데이터 표시 로직)는 이 명세 범위 밖이며, 각 위젯은 독립적으로 개발된 컴포넌트를 사용한다.
- 터치 기반 드래그(모바일)는 Phase 1에서는 고려하지 않으며, 데스크탑/마우스 환경을 우선 지원한다.
- 동일 사용자가 여러 탭에서 동시에 대시보드를 편집하는 시나리오는 고려하지 않는다.
- 위젯 정의 목록은 개발자가 코드에 정적으로 등록하며, 관리자 UI를 통한 동적 등록은 이번 범위 밖이다.
